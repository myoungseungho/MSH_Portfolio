Part 1. 개발자는 일반 사용자와 뭐가 다른가

기획자가 도구를 받았을 때와 개발자가 받았을 때, 기대가 같을까?

다르다. 개발자는 내부가 어떻게 돌아가는지도 알고 싶어한다.

대상 도구에 대한 태도
일반 사용자 블랙박스여도 됨. 결과만 나오면 OK
개발자 내부가 어떻게 돌아가는지 알고 싶어함

Part 2. "투명"의 두 가지 의미

UX에서는 도구가 "투명해야" 한다고 했는데, 개발자는 내부를 보고 싶어한다면 모순 아닌가?

같은 단어가 반대 의미로 쓰인다.

"Transparent"의 두 의미

맥락 의미 원하는 것
UX 투명성 보이지 않는다 (유리처럼) 도구가 사라지길
DX 투명성 들여다볼 수 있다 내부가 보이길
둘 다 만족시킬 수 있나?

가능하다. Progressive Disclosure를 적용하면:

- 평소엔 마찰 없이 작동 (UX 투명성)

- 필요할 때 내부를 볼 수 있음 (DX 투명성)

# 평소 - 결과만 mytool run # 문제 생겼을 때 - 내부가 보임 mytool run --verbose mytool run --debug

Part 3. Inspectability (검사 가능성)

Inspectability

내부 상태를 들여다볼 수 있는 능력

개발자는 "필요하면 볼 수 있다"는 사실 자체로 신뢰한다.

방법 예시
로그 레벨 info, debug, verbose
상태 확인 mytool status
설정 덤프 mytool config --show
드라이런 mytool run --dry-run

Part 4. Time to Hello World

개발자가 새 도구를 받았을 때, "이거 쓸만하네"라고 느끼는 순간은?

설치하고 바로 돌아갈 때.

Time to Hello World

도구를 받아서 처음으로 뭔가 작동하는 걸 보기까지의 시간

용어 대상 목표 지점
Time to Value 일반 사용자 "이거 유용하네"
Time to Hello World 개발자 "일단 돌아가네"
왜 이게 중요해?

개발자는 새 도구를 의심부터 한다.

"내 환경에서 돌아갈까?", "의존성 충돌 안 나려나?"

이 의심이 "일단 돌아간다"를 보는 순간 풀린다.

Part 5. Zero Configuration

Zero Configuration / Convention over Configuration

설정 없이도 일단 돌아가게 하라

설정은 기본 규칙을 벗어날 때만 하게 하라

서버 IP/포트가 7~8개 필요하면 어떻게 해?

프리셋을 만든다:

[처음 실행] 환경을 선택하세요: ● 개발 서버 (dev) ○ 테스트 서버 (test) ○ 라이브 서버 (live) ○ 직접 설정 → 한 번 선택으로 7~8개 설정 완료

Configuration Layering (설정 계층화)

계층 역할 예시
프리셋 80%를 커버 dev, test, live
개인 설정 나머지 20% 내 로컬 DB만 변경
임시 오버라이드 일회성 --port 8080

Part 6. Actionable Error (행동 가능한 에러)

일반 사용자용 에러 메시지를 개발자가 보면 뭐라고 할까?

"더 자세히 알려달라"

일반 사용자용 vs 개발자용 에러

❌ 일반 사용자용만 있으면: 서버에 연결할 수 없습니다. ✅ 개발자용: [ERROR] Connection failed to GameDB server Host: 192.168.0.100:3306 Error: ETIMEDOUT (10060) Timeout: 5000ms Possible causes: - Server not running - Firewall blocking port 3306 - Wrong IP/Port in config Debug: mytool --verbose 로 상세 로그 확인 Config: C:\mytool\config.json (line 12)
요소 내용
Where 어디서 실패? (IP, 포트, 파일 경로)
What 정확히 무슨 에러? (에러 코드)
Why 가능한 원인들
How 다음에 뭘 해볼까? (디버그 명령, 설정 위치)

Part 7. 에러 메시지 계층화

기획자도 쓰고 개발자도 쓴다면, 에러를 어떻게 보여줘야 해?

계층화한다:

기본 모드 (기획자용) ├─ "서버에 연결할 수 없습니다" │ --verbose 모드 (개발자용) ├─ 상세 에러 코드, IP, 포트 ├─ 가능한 원인 목록 │ 로그 파일 (항상 기록) └─ 전체 스택트레이스, 타임스탬프
계층 누가 언제
기본 모든 사용자 항상
verbose 개발자 문제 해결할 때
로그 파일 개발자 나중에 분석할 때

핵심 패턴

"화면엔 간단히, 로그엔 자세히"

기획자가 봐도 당황하지 않고, 개발자는 필요하면 파고들 수 있다.

Part 8. Just-in-time Documentation

개발자가 새 도구를 받으면 README를 먼저 볼까?

아니. 일단 실행부터 해본다.

Documentation Paradox (문서화의 역설)

"개발자는 문서를 안 읽는다.

하지만 문서가 없으면 화낸다."

실제 행동 패턴: 1. 일단 실행해본다 2. 안 되면 에러 메시지를 본다 3. 그래도 모르면 --help를 친다 4. 그래도 안 되면 README 5. 최후의 수단: 구글링

Just-in-time Documentation

문서는 막힌 순간에 나타나야 한다

막힌 순간 문서가 나타나는 곳
에러 발생 에러 메시지 안에 해결책
명령어 모름 --help
옵션 모름 mytool cmd --help
설정 형식 모름 에러에 예시 파일 경로

Part 9. 좋은 --help 설계

$ mytool --help Usage: mytool <command> [options] Commands: run 서버 실행 status 현재 상태 확인 config 설정 관리 Options: -v, --verbose 상세 로그 출력 -h, --help 도움말 보기 Examples: mytool run # 기본 실행 mytool run --verbose # 상세 로그와 함께 mytool config --show # 현재 설정 확인 Documentation: https://github.com/...

핵심: Examples

개발자는 설명보다 예시를 보고 이해한다.


"Show, don't tell"

설명하지 말고 보여줘라

❌ "run 명령어는 서버를 실행합니다" ✅ "mytool run --port 8080"

핵심 정리: DX 원칙

  • DX 투명성 - 개발자는 내부를 들여다보고 싶어한다
  • Inspectability - 볼 수 있다는 사실 자체가 신뢰
  • Time to Hello World - "일단 돌아가네"까지의 시간을 줄여라
  • Zero Configuration - 설정 없이 일단 돌아가게
  • Configuration Layering - 프리셋 → 개인설정 → 임시 오버라이드
  • Actionable Error - Where, What, Why, How를 담아라
  • 에러 계층화 - 화면엔 간단히, 로그엔 자세히
  • Documentation Paradox - 안 읽지만 없으면 화낸다
  • Just-in-time Docs - 막힌 순간에 문서가 나타나야
  • Show, don't tell - 설명보다 예시
