Part 1. 도구란 무엇인가

"툴"과 "앱"과 "프로그램"은 뭐가 달라?

셋 다 실행 가능한 소프트웨어인데, 왜 어떤 건 "툴"이라 부르고 어떤 건 "앱"이라 부를까?

구분 특징 예시
앱/프로그램 그 자체가 목적 넷플릭스를 "쓰려고" 연다
툴 다른 목적을 위한 수단 뭔가를 "만들려고" 연다
그게 왜 중요해?

이 차이가 설계 철학을 완전히 바꾸기 때문이야.

앱은 "머무르게" 해야 하고, 툴은 "빨리 끝내게" 해야 해.

Part 2. 도구의 투명성 - 하이데거

망치를 쓸 때, 망치 자체에 대해 얼마나 생각해?

못을 박을 때 집중하는 건 "못이 잘 들어가는가"이지,

"망치의 무게 배분이 어떻게 되어있지?"가 아니잖아.

그런데 망치 손잡이가 미끄럽다면?

그 순간 집중이 "못"에서 "망치"로 옮겨가.

하려던 일에서 벗어나 도구 자체를 의식하게 돼.

도구의 투명성 (Tool Transparency)

마르틴 하이데거 (Martin Heidegger)
상태 독일어 의미
손안의 존재 Zuhandenheit 도구가 손에 익어서 의식하지 않음
눈앞의 존재 Vorhandenheit 도구가 고장나서 갑자기 "보이기" 시작함

좋은 도구는 "투명해져야" 한다. 사용자가 도구를 의식하지 않고 오직 하려는 일에만 집중할 수 있어야 한다.

소프트웨어에서 "눈앞의 존재"가 되는 순간은 언제야?

두 가지 상황:

1. 처음 설치/실행 - 아직 손에 익지 않은 상태

2. 오류 발생 - 손에 익었다가 깨진 상태

Part 3. Time to Value - 가치 도달 시간

사용자가 exe를 더블클릭했어. 이 사람의 목표는 뭐야?

"이 앱을 구경하고 싶다"가 아니라

"빨리 내 일을 하고 싶다"야.

Time to Value (TTV)

"처음 실행"부터 "아, 이거 쓸만하네"라고 느끼는 순간까지의 시간.

그 순간을 Aha Moment라고 부른다.


TTV가 길면 → 사용자가 포기

TTV가 짧으면 → 사용자가 정착

그러면 첫 화면에서 설정을 다 해야 한다면?

TTV가 길어지겠지. 여기서 중요한 개념:

Sensible Defaults (합리적 기본값)

"80%의 사용자에게 맞는 값을 미리 넣어둬라"

대부분은 설정 없이 바로 쓸 수 있고,

특수한 경우만 나중에 바꾸면 된다.

Part 4. Affordance와 Signifier

문고리를 보면 "당겨야겠다" 또는 "밀어야겠다"고 느끼잖아. 왜?

생긴 모양이 행동을 유도하기 때문이야.

Affordance (행동유도성)

사물이 "이렇게 써라"고 스스로 알려주는 성질

제임스 깁슨 → 도널드 노먼 《The Design of Everyday Things》

사물 Affordance
손잡이가 튀어나온 문 "당겨라"
평평한 판이 달린 문 "밀어라"
버튼 "눌러라"
소프트웨어는? 버튼은 실제로 눌리는 게 아니잖아.

그림자, 색상 변화로 "눌릴 것처럼" 보이게 만들어.

이건 진짜 Affordance가 아니라 Signifier야.

Affordance vs Signifier

개념 의미 예시
Affordance 실제로 가능한 것 문은 열 수 있다
Signifier 가능함을 알려주는 신호 "당기시오" 표지판

소프트웨어는 진짜 Affordance가 없으니 Signifier에 의존한다.

Part 5. Mental Model과 관습

요즘 버튼은 납작한데, 왜 여전히 클릭해야 할 것 같지?

학습된 거야. 수십 년간 버튼을 클릭해왔으니까.

Mental Model (심성 모델)

사용자가 이전 경험을 통해 갖는 기대

"이렇게 생긴 건 이렇게 작동할 것"

Jakob's Law (야콥의 법칙)

"사용자는 대부분의 시간을 다른 서비스에서 보낸다.

그래서 당신의 서비스도 다른 것과 비슷하게 작동하길 기대한다."


→ 새로운 걸 발명하지 마라

→ 관습(Convention)을 따라라

그럼 혁신은 언제 해?

기존 습관을 버리게 하려면, 새로운 방식이 10배는 더 좋아야 해.

자동차 비유

모든 자동차는 핸들, 브레이크 위치가 같다. (관습)
하지만 엔진 성능, 연비는 다르다. (혁신)

"어떻게 쓰는가"는 관습을 따르고,
"무엇을 해주는가"에서 차별화하라.

Part 6. Cognitive Load와 선택의 역설

Cognitive Load (인지 부하)

사용자의 뇌가 한 번에 처리할 수 있는 정보량에는 한계가 있다.


마법의 숫자 7±2 (조지 밀러, 1956)

→ 사람은 단기 기억에 5~9개 항목만 유지할 수 있다.

Hick's Law (힉스 법칙)

선택지가 많을수록 결정 시간이 길어진다

선택지 2개 → 빠르게 선택

선택지 20개 → 고민하다 포기

기능이 많은데 다 보여주면?

사용자가 질린다. 안 보여주면 기능을 못 찾는다.

해결책:

Progressive Disclosure (점진적 노출)

필요한 순간에, 필요한 만큼만 보여준다


단계 보여주는 것
첫 화면 가장 많이 쓰는 1~2개
한 단계 들어가면 자주 쓰는 5~6개
더 들어가면 전문가용 전체 옵션

예시: 구글 첫 화면은 검색창 하나. 하지만 "고급 검색"에는 수십 개 옵션.

Part 7. Feedback - 반응의 원칙

버튼을 클릭했는데 3초간 아무 반응이 없다면?

"클릭이 안됐나? 다시 눌러볼까?"

→ 중복 요청 발생, 짜증, 이탈

Feedback 원칙

모든 행동에는 즉각적인 반응이 있어야 한다

상황 피드백 예시
클릭 즉시 버튼 색 변화, 눌림 효과
처리 중 로딩 스피너, 진행률 표시
완료 "저장되었습니다" 메시지
실패 에러 메시지 + 해결 방법

Response Time (응답 시간 법칙)

야콥 닐슨

시간 사용자 인식
0.1초 즉각 반응으로 느낌
1초 지연은 느끼지만 흐름 유지
10초 집중력 한계, 이탈 시작

Part 8. Error Prevention - 실수 방지

이 두 에러 메시지의 차이는?

Error: 0x80040E10 → 개발자용

서버에 연결할 수 없습니다. 네트워크를 확인해주세요. → 사용자용

좋은 에러 메시지의 3요소

요소 설명 예시
What 무슨 일이 일어났는가 "서버에 연결할 수 없습니다"
Why 왜 일어났는가 "네트워크가 불안정합니다"
How 어떻게 해결하는가 "[네트워크 확인] 버튼 클릭"
에러보다 더 좋은 건?

에러가 안 나는 것.

Error Prevention (에러 방지)

사용자가 실수하기 어렵게 만들어라

방법 설명 예시
제약 (Constraint) 잘못된 입력 자체를 막음 날짜 필드에 달력만 뜨게
경고 (Warning) 위험한 행동 전에 알림 "정말 삭제하시겠습니까?"
기본값 (Default) 흔한 실수를 미리 방지 안전한 옵션이 기본 선택
되돌리기 (Undo) 실수해도 복구 가능 Gmail "전송 취소"

핵심 정리: 개발자를 위한 UX 원칙

  • 도구의 투명성 - 좋은 도구는 의식되지 않는다 (하이데거)
  • Time to Value - 가치 도달 시간을 최소화하라
  • Sensible Defaults - 80%에게 맞는 기본값을 넣어라
  • Affordance/Signifier - 사용법이 보이게 만들어라
  • Mental Model - 사용자의 기존 경험을 존중하라
  • Jakob's Law - 익숙한 패턴을 따르라
  • Hick's Law - 선택지를 줄여라
  • Progressive Disclosure - 필요한 순간에 필요한 만큼만
  • Feedback - 모든 행동에 즉각 반응하라
  • Error Prevention - 실수하기 어렵게 만들어라
