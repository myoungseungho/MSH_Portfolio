AI 에이전트 CEO론

일반 AI vs 에이전트의 차이, 그리고 AI 에이전트 팀의 CEO가 되는 방법

2025.01 · Q&A 학습 기록

Part 1. 일반 AI vs 에이전트

GPT한테 "이메일 10개 보내줘"라고 하면 뭐가 나와?

그냥 텍스트로 예시만 보여주지, 실제로 안 보내.

"이런 내용으로 보내면 됩니다~" 하고 끝.

왜 실제로 못 보내?

도구가 없어서.

일반 AI는 생각하고 말하는 것까지만 해. 실제로 이메일 시스템에 접근하는 도구가 없으면 보낼 수가 없어.

그럼 에이전트는 뭐가 달라?

도구 실행 + 행동을 해.

생각만 하는 게 아니라, 실제로 도구를 써서 행동한다.

일반 AI: 입력 → 생각 → 텍스트 출력 (끝) 에이전트: 입력 → 생각 → 도구 실행 → 행동
근데 일반 AI도 가끔 "보내준다" 기능이 있잖아. 그거랑 뭐가 달라?

맞아, 일반 AI도 도구가 붙으면 보내긴 해.

근데 핵심 차이는 "실패하면?"이야.

  • 일반 AI: 실패하면 "실패했습니다" 하고 끝
  • 에이전트: 실패하면 → 원인 분석 → 다시 시도 → 될 때까지
될 때까지? 그게 뭐가 중요해?

이게 바로 "자율성"이야.

에이전트 루프: 입력 → 생각 → 행동 → 결과 확인 ↓ 실패? → 다시 생각 → 다시 행동 ↓ 성공할 때까지 반복

사람이 중간에 개입 안 해도 알아서 목표를 달성할 때까지 시도한다.

그건 위험하지 않아?

위험해.

예를 들어 "이메일 보내줘" 했는데 에이전트가 이상한 내용으로 CEO한테 메일 보내면?

자율성이 높을수록 → 리스크도 높아.

그럼 어떻게 해야 해?

에이전트에게 권한과 범위를 정해줘야 해.

  • 뭘 해도 되는지
  • 뭘 하면 안 되는지
  • 어디까지 자율적으로 하고, 어디서 사람한테 물어볼지
핵심 정리

일반 AI = 입력을 하면 생각을 해서 자기 생각만 얘기한다

에이전트 = 그걸 행동한다. 될 때까지 행동한다

Part 2. 왜 에이전트가 여러 개 필요해?

에이전트 하나가 모든 일을 다 할 수 있을까?

코드 작성, 코드 리뷰, 테스트 실행, 문서 작성, 버그 수정...

이걸 에이전트 하나한테 전부 시키면?

에이전트도 입력 → 생각 → 행동 → 검토 → 생각 → 행동 계속 이러는데, 마치 싱글스레드처럼 일손이 부족해.

싱글스레드 병목은 어떻게 해결해?

멀티스레드 쓰지.

비유

싱글스레드 → 멀티스레드

에이전트 하나 → 에이전트 여러 개

멀티스레드 쓸 때 스레드들이 막 돌아가게 했어?

아니, 메인 스레드가 관리했지.

  • 메인 스레드: 일 분배, 결과 수집, 조율
  • 워커 스레드: 실제 작업 수행
에이전트도 그렇게 구조화할 수 있어?

똑같아.

메인 에이전트 (플래너/오케스트레이터) = 일 분배, 결과 검토, 조율 서브 에이전트들 = 실제 작업 수행

Part 3. AI 에이전트 팀의 CEO가 되기

그럼 나는 어디에 있고 싶어?

메인 에이전트 자리? 서브 에이전트 자리? 아니면 그 위?

그 위가 바로 CEO 자리야.

너 (CEO) ↓ 지시 메인 에이전트 (팀장/플래너) ↓ 분배 서브 에이전트들 (실무자)
CEO가 하는 일이 뭐야?

직접 코드 짜? 직접 테스트 돌려? 아니.

지시하고 검토하지.

  • What 정의 - "이거 해"
  • 결과 검토 - "이거 맞아?"
  • 승인/반려 - "OK" 또는 "다시 해"

How는 안 해. 에이전트들이 알아서 해.

CEO가 서브 에이전트 100명한테 직접 지시하면?

힘들지. 회사도 CEO가 직원 100명 직접 관리 안 해.

중간에 중간 관리자가 있어. CEO는 임원진들끼리만 독대하지.

재귀적으로 팀들이 구성돼.

CEO (너) ↓ 임원진 (플래너, 리뷰어) ↓ 팀장들 ↓ 실무자들
CEO는 누구랑 대화해?

메인 에이전트 = 비서실장 같은 존재.

너 (CEO) ↓ ← 여기만 직접 대화 메인 에이전트 (비서실장) ↓ ← 비서실장이 알아서 관리 서브 에이전트들 + 스킬들

Part 4. 서브에이전트 vs 스킬

"서브에이전트"랑 "스킬" 구분이 뭐야?
  • 서브에이전트 = 정규직
  • 스킬 = 프리랜서
정규직이랑 프리랜서 차이가 뭐야?

정규직은 생각하고, 프리랜서는 시키는 것만 하지.

서브에이전트 (정규직) 스킬 (프리랜서)
특징 생각한다, 판단한다 시키는 것만 한다
예시 "이 버그 고쳐" → 알아서 원인 찾고 수정 "git commit 해" → 딱 그것만 함
비용 비쌈 (토큰 많이 씀) 쌈 (토큰 적게 씀)
속도 느림 빠름
왜 둘 다 필요해? 전부 정규직으로 하면 안 돼?

정규직은 비싸잖아.

회사 비유

전략 짜는 일 → 정규직 (비싸도 필요)

택배 보내기 → 프리랜서/외주 (굳이 정규직 쓸 필요 없음)

Part 5. CEO의 원칙: What > How

좋은 지시 vs 나쁜 지시 차이가 뭐야?

지시 A: "로그인 기능 만들어"

지시 B: "로그인 기능 만들어. Express 써서 /api/login 엔드포인트 만들고, bcrypt로 비밀번호 해싱하고, JWT 토큰 발급해서 리턴해"

CEO로서 어떤 게 더 좋아?

B가 더 자세하니까 좋은 거 아니야?

아니. CEO는 A가 맞아.

CEO가 How까지 지시하면?

  • = 마이크로매니징
  • = 비서실장/팀장 존재 의미 없음
  • = 비효율
CEO: "로그인 기능 만들어" (What만) ↓ 팀장: "Express vs Fastify? JWT vs Session?" → 대안 검토 → 결정 → 지시 ↓ 실무자: 실제 구현
What > How 철학

CEO는 What만 말한다

How는 에이전트들이 알아서 한다

Part 6. 검토와 견제 (삼권분립)

에이전트가 잘못된 방향으로 가면 어떡해?

검토하고 다시 시키면 되지.

근데 그 검토는 CEO가 해? 아니면 비서실장한테 시켜?

검토도 위임할 수 있어?

검토도 두 종류가 있어:

종류 누가 예시
실무 검토 리뷰어 에이전트 코드 품질, 버그, 스타일
전략 검토 CEO (너) 방향이 맞는지, 최종 승인

CEO가 모든 검토 다 하면? = CEO가 병목 = 일 안 돌아감

삼권분립처럼 견제 구조를 만들면 더 단단해질까?
에이전트 삼권분립: 플래너 (입법) ─ 계획 수립, 무엇을 할지 ↓ 실행자 (행정) ─ 실제 작업 수행 ↓ 리뷰어 (사법) ─ 결과 검토, 견제 ↓ 문제 있으면 → 플래너에게 피드백 → 재계획

맞아, 더 단단해져. 하지만 트레이드오프가 있어:

견제 없음 견제 있음
속도 빠름 느림
비용 쌈 비쌈
품질 실수 가능 더 정확
안정성 낮음 높음
그럼 언제 견제를 넣어야 해?

중요한 것만.

견제가 필요한 경우:

  • 배포/출시
  • 결제/돈 관련
  • 보안/인증
  • 삭제/되돌릴 수 없는 작업

견제 없이 해도 되는 경우:

  • 파일 읽기
  • 간단한 검색
  • 임시 테스트

Part 7. Pull vs Push - 진짜 CEO처럼 일하기

지금 에이전트랑 소통 방식이 어때?

콘솔 열고 → 명령 치고 → 기다리고 → 결과 받고

내가 먼저 물어봐야 대답이 와.

이걸 Pull 방식이라고 해.

진짜 회사 CEO는 어때?

매번 직원한테 "일 어떻게 돼?" 하고 물어봐?

아니. 직원이 알아서 보고해.

이게 Push 방식이야.

Pull: 너 → "일 어떻게 돼?" → 에이전트 (네가 물어봄) Push: 에이전트 → "보고드립니다" → 너 (알아서 옴)
Push 방식으로 바꾸려면 뭐가 필요해?

에이전트가 "알아서" 보고하려면, 언제 보고해야 하는지 알아야 해.

그 "언제"를 트리거라고 불러.

트리거 종류가 뭐가 있어?
트리거 종류 예시
시간 기반 매일 아침 9시 "일일 보고입니다"
이벤트 기반 에러 발생 "긴급! 서버 죽었습니다"
완료 기반 작업 끝 "배포 완료했습니다"
핵심

Pull (현재): CEO가 물어봐야 답이 옴 = 관리자처럼 일함

Push (목표): 에이전트가 알아서 보고 = 진짜 CEO처럼 일함

Part 8. 자동화 시스템 설계

보고를 어디로 받고 싶어?

콘솔? 아니면 자주 쓰는 메신저?

디스코드가 좋아:

  • 완전 무료
  • 봇 API 쉬움
  • 채널 구조 = 부서 구조
디스코드가 단순 알림만?

아니. 업무 공간이 될 수 있어.

기존 생각: 디스코드 = 알림 받는 곳 새로운 생각: 디스코드 = 에이전트들의 사무실

에이전트들이 분석한 문서, 대안 검토 내용 다 채널에 올라감.

채널 구조는 어떻게?
디스코드 서버 = 회사 채널 = 부서/팀 #ceo-보고 ← 취합본 (핵심만) #코드리뷰 ← 디테일 #빌드결과 ← 디테일 #아이디어 ← 디테일 ...
CEO는 뭘 봐?

평소: #ceo-보고 만 봄 (취합본)

가끔: 각 채널 들어가서 디테일 확인

= 정보의 계층화

보고 싶으면 볼 수 있고, 안 보고 싶으면 안 봐도 돼.

에이전트가 24시간 돌아가려면?

24시간 서버가 필요해.

24시간 서버에서 돌아가는 것: ├─ 디스코드 봇 → 채널이랑 소통 (받고/올리고) ├─ Claude API → 실제로 생각하고 행동 └─ 트리거 시스템 → 언제 실행할지
흐름이 어떻게 돼?

예: "매일 아침 9시 보고"

1. 아침 9시 됨 (트리거) 2. 봇이 Claude API 호출 "오늘 보고할 거 정리해" 3. Claude가 생각하고 답변 4. 봇이 그 답변을 #ceo-보고 채널에 올림 5. CEO는 핸드폰으로 디스코드 알림 받음

Part 9. 최종 설계 구조

전체 그림이 어떻게 돼?
너 (CEO) │ ├─ 📱 디스코드 (사무실) │ ├─ #ceo-보고 (취합본) │ ├─ #코드리뷰 │ ├─ #빌드결과 │ ├─ #아이디어 │ └─ ... │ ├─ 🖥️ 24시간 서버 │ ├─ 디스코드 봇 │ └─ Claude API 연동 │ └─ ⏰ 트리거 ├─ 시간 기반 (매일 아침) ├─ 이벤트 기반 (에러 발생) └─ 완료 기반 (작업 끝)
CEO가 아직 정해야 할 것은?
  • 뭘 감시할지 - 코드 변경? 빌드? 시간?
  • 언제 보고받을지 - 매일? 매주? 이벤트 발생 시?
  • 어떤 채널 구조로 할지 - 어떤 부서가 필요?
  • 첫 번째 자동화할 업무가 뭔지

이게 CEO의 일이야. What을 정의하는 것.

AI 시대의 CEO

호기심 많은 사람 = What을 많이 정의할 수 있는 사람

How 전문가들은 점점 에이전트로 대체돼.

What 정의하는 사람은 대체 안 돼.

총정리: AI 에이전트 CEO론

일반 AI vs 에이전트

  • 일반 AI: 생각하고 말한다
  • 에이전트: 생각하고 행동한다, 될 때까지

에이전트 회사 구조

CEO (너) ├─ What만 지시 ├─ 전략 검토 └─ 최종 승인 플래너 (비서실장) ─ 일 분배, 조율 리뷰어 (품질팀장) ─ 실무 검토 서브에이전트 (정규직) ─ 생각하는 일, 비쌈 스킬 (프리랜서) ─ 실행만, 쌈

CEO 원칙

  • What > How - 마이크로매니징 금지
  • 검토 위임 - CEO가 병목 되면 안 됨
  • 중요한 것만 직접 - 최종 승인, 큰 결정

CEO가 정해야 할 것들

  • What - 뭘 할지
  • 권한 범위 - 뭘 해도 되고 안 되는지
  • 견제 수준 - 어떤 작업에 검토 넣을지
  • 최종 승인 - 중요한 건 직접 확인

자동화 시스템

  • Pull → Push - 물어보는 게 아니라 보고받기
  • 트리거 - 시간/이벤트/완료 기반
  • 디스코드 - 에이전트들의 사무실
  • 24시간 서버 - 봇 + Claude API

AI 시대의 가치

  • 호기심 많은 사람 = What을 정의하는 사람
  • How 전문가는 에이전트로 대체됨
  • What 정의자는 대체 안 됨
