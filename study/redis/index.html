Redis vs MSSQL 실시간 랭킹 성능 비교

C++ / Windows Docker 환경
Redis MSSQL Sorted Set 성능 테스트 게임 서버

개요

게임 서버에서 실시간 랭킹 시스템을 구현할 때, RDBMS와 Redis 중 어떤 것이 더 적합한지 직접 성능 테스트를 통해 비교해보았습니다.

두 가지 시나리오를 테스트했습니다:

  1. 실제 게임 서버 환경: 21개 격리된 DB에서 전체 랭킹 취합
  2. 공정한 비교: 동일 조건(인덱스/Sorted Set)에서 순수 성능 비교

테스트 환경 배경: 실제 게임 서버 구조

실제 MMO RPG 게임 서버는 다음과 같은 구조로 운영됩니다:

┌─────────────────── 1번 서버군 ───────────────────┐ │ 채널 1-1 채널 1-2 채널 1-3 │ │ (1000명) (1000명) (1000명) │ │ ↓ ↓ ↓ │ │ DB 1-1 DB 1-2 DB 1-3 │ │ (격리됨) (격리됨) (격리됨) │ └─────────────────────────────────────────────────┘ ... 2~6번 서버군 ... ┌─────────────────── 7번 서버군 ───────────────────┐ │ 채널 7-1 채널 7-2 채널 7-3 │ │ ↓ ↓ ↓ │ │ DB 7-1 DB 7-2 DB 7-3 │ └─────────────────────────────────────────────────┘ 총: 7 서버군 x 3 채널 = 21개 격리된 DB 각 채널당 1000명 = 총 21,000명

왜 DB가 격리되어 있나?

일부 MMO RPG에서는 채널별로 독립적인 DB를 운영합니다. 같은 채널 내에서만 유저가 상호작용하고, 전체 랭킹 같은 기능에서만 모든 DB를 취합하는 구조입니다. (게임마다 구조는 다를 수 있음)

테스트 환경

항목 상세
OS Windows (Docker Desktop)
Redis Docker Container (localhost:6379)
MSSQL Docker Container (localhost:1433)
언어 C++ (Visual Studio 2022)
Redis 연결 TCP Socket 직접 구현 (Winsock)
MSSQL 연결 ODBC

Docker 컨테이너 실행

# Redis 실행
docker run --name my-redis -d -p 6379:6379 redis

# MSSQL 실행
docker run --name my-mssql \
  -e "ACCEPT_EULA=Y" \
  -e "SA_PASSWORD=Test1234!" \
  -p 1433:1433 \
  -d mcr.microsoft.com/mssql/server:2019-latest

Test 1: 실제 게임 서버 환경 (21개 DB 취합)

시나리오

  • 7개 서버군 x 3개 채널 = 21개 격리된 테이블
  • 각 채널당 1000명 = 총 21,000명
  • 각 테이블에 INDEX(score DESC) 적용
  • 전체 랭킹 조회: 21개 테이블에서 데이터 취합 필요

MSSQL 방식 (전체 랭킹 조회 시)

전체 랭킹 요청! │ ▼ ┌─────────────────────────────────────────────────┐ │ 21개 DB 순회 │ │ │ │ 1. Channel_1_1: SELECT TOP 100 ORDER BY score │ │ 2. Channel_1_2: SELECT TOP 100 ORDER BY score │ │ ... (21번 반복) │ │ 21. Channel_7_3: SELECT TOP 100 ORDER BY score │ │ │ │ → 2100개 결과 수집 │ │ → 메모리에서 다시 정렬 │ │ → TOP 100 추출 │ └─────────────────────────────────────────────────┘

Redis 방식

전체 랭킹 요청! │ ▼ ┌─────────────────────────────────────────────────┐ │ Redis 중앙 Sorted Set │ │ │ │ ZREVRANGE global_ranking 0 99 │ │ │ │ → 이미 정렬된 TOP 100 즉시 반환 │ └─────────────────────────────────────────────────┘ (점수 변경 시 각 채널에서 Redis에 ZADD로 동기화)

Test 1 결과: 전체 랭킹 조회 100회 (3회 측정)

회차 MSSQL Redis 배율
1차 2,519ms 34ms 74배
2차 2,527ms 27ms 93배
3차 2,581ms 49ms 52배
Redis 약 50~90배 빠름!

* MSSQL은 일정(~2.5초), Redis는 네트워크/시스템 상태에 따라 변동

Test 2: 공정한 비교 (동일 조건)

왜 공정한 비교가 필요한가?

Test 1은 "21개 테이블 취합"이라는 추가 오버헤드가 있었습니다. MSSQL도 단일 테이블 + 인덱스를 사용하면 정렬 상태를 유지할 수 있습니다. 순수하게 저장소 차이만 비교해봅니다.

동일 조건

MSSQL Redis
테이블/Set GlobalRanking 1개 Sorted Set 1개
정렬 유지 INDEX (B-Tree) Skip List
업데이트 UPDATE 1번 (자동 정렬) ZADD 1번 (자동 정렬)
조회 SELECT TOP 100 ZREVRANGE
유저 수 21,000명 21,000명

테스트 내용

1000번 반복:

  1. 랜덤 유저 점수 변경 (UPDATE / ZADD)
  2. TOP 100 조회 (SELECT / ZREVRANGE)

Test 2 결과: 업데이트 + 조회 1000회

방식 총 시간 평균
MSSQL (INDEX 있음) 2,534ms 2.53ms
Redis (Sorted Set) 716ms 0.72ms
Redis 3.5배 빠름!

왜 3.5배 차이인가?

둘 다 O(log N) 알고리즘을 사용하지만, 차이는 저장소입니다:

  • MSSQL: 디스크 기반 → 디스크 I/O 발생
  • Redis: 메모리 기반 → 메모리 접근만

결과 요약

테스트 시나리오 Redis 우위
Test 1 21개 테이블(INDEX 있음) 취합 vs Redis 50~90배
Test 2 인덱스 있는 단일 테이블 vs Redis 3.5배

핵심 인사이트

공정한 비교(Test 2)에서도 Redis가 3.5배 빠릅니다.

하지만 실제 게임 서버에서 전체 랭킹을 구현하려면 MSSQL은 "21개 테이블 취합" 구조가 될 수밖에 없고, 인덱스가 있어도 50~90배 차이가 납니다. (테이블당 정렬은 빨라도, 취합+재정렬 오버헤드가 큼)

핵심 개념: 정렬 시점의 차이

MSSQL (인덱스 없을 때)

조회할 때마다 정렬

점수 변경 → 저장만 점수 변경 → 저장만 점수 변경 → 저장만 ... 랭킹 조회 → 정렬! (오래 걸림)

Redis (Sorted Set)

변경할 때마다 정렬

점수 변경 → 저장 + 정렬 점수 변경 → 저장 + 정렬 점수 변경 → 저장 + 정렬 ... 랭킹 조회 → 바로 반환! (빠름)

게임에서 점수 변경은 가끔(몬스터 잡을 때, 퀘스트 깰 때), 랭킹 조회는 자주(버튼 누를 때마다) 일어납니다. → 조회를 빠르게 만드는 Redis 방식이 효율적!

Redis Sorted Set이란?

Redis는 단순한 Key-Value 저장소가 아니라, 여러 자료구조를 지원합니다:

자료구조 설명 예시
String 단순 값 SET name "철수"
Hash 필드-값 쌍 HSET user:1 name "철수"
List 순서 있는 목록 LPUSH queue "작업1"
Set 중복 없는 집합 SADD online "철수"
Sorted Set 점수로 자동 정렬 ZADD ranking 1000 "철수"

명령어 해석

ZADD ranking 1000 "철수"
 │     │      │     │
 │     │      │     └─ member (누구?)
 │     │      └─────── score (몇 점?)
 │     └────────────── key (어떤 랭킹?)
 └──────────────────── 명령어 (Sorted Set에 추가)

ZREVRANGE ranking 0 99
 │          │     │  │
 │          │     │  └─ 99번째까지 (100개)
 │          │     └──── 0번째부터
 │          └────────── key 이름
 └───────────────────── 높은 점수순 범위 조회

결론

언제 Redis를 쓰는가?

  • 읽기가 매우 빈번한 데이터 - 세션, 설정값, 캐시
  • 실시간 순위/정렬이 필요한 데이터 - 랭킹, 리더보드
  • 분산 서버 간 공유가 필요한 데이터 - 파티 정보, 매칭 큐
  • TTL(만료)이 필요한 데이터 - 임시 토큰, 쿨다운

언제 RDBMS를 쓰는가?

  • 영구 저장이 필요한 데이터 - 유저 정보, 인벤토리
  • 복잡한 쿼리/JOIN이 필요한 경우 - 통계, 리포트
  • 트랜잭션 보장이 필요한 경우 - 결제, 거래
  • 데이터 무결성이 중요한 경우 - 마스터 데이터

핵심 교훈

Redis와 RDBMS는 경쟁 관계가 아니라 상호 보완 관계입니다.
실시간/캐시 데이터는 Redis에, 영구/복잡한 데이터는 RDBMS에 저장하는 것이 일반적인 게임 서버 아키텍처입니다.
