SSL/TLS: 인터넷 보안의 기초

초보자를 위한 개념 정리 2025.01
SSL TLS HTTPS 인증서 암호화

개요

서버를 세팅하다 보면 "SSL 인증서 설치"라는 말을 듣게 됩니다. SSL이 뭔지, 왜 필요한지, 어떻게 동작하는지를 질문과 답변 형식으로 차근차근 이해해봅시다.

이 문서는 "비밀 편지를 어떻게 안전하게 보낼까?"라는 아주 기초적인 질문에서 시작해서, SSL의 전체 그림을 그려나갑니다.

1. 암호화란? - 비밀 편지 문제

질문

친구에게 비밀 편지를 보내려고 합니다. 그런데 다른 사람이 중간에 가로채서 읽을 수 있는 상황이라면, 어떻게 하면 친구만 내용을 알 수 있게 할까요?

답

둘만 아는 암호로 내용을 바꿔서 보낸다!

이게 바로 암호화(Encryption)의 핵심 아이디어입니다. 둘만 아는 규칙(암호)으로 내용을 섞어서 보내고, 받는 사람은 그 규칙을 알기 때문에 다시 원래대로 풀 수 있는 거죠.

보내는 사람 받는 사람 │ │ "안녕하세요" ─── 암호화 ───> "X#@$%" ─── 복호화 ───> "안녕하세요" │ │ 둘만 아는 같은 규칙으로 규칙 적용 해독

이렇게 같은 키(규칙)로 암호화하고 복호화하는 방식을 대칭키 암호화라고 부릅니다.

2. 키 교환 문제 - 딜레마 발생

질문

처음 보는 사람(예: 인터넷 쇼핑몰)에게 비밀 정보(카드번호)를 보내야 합니다. "둘만 아는 암호"를 쓰고 싶은데... 그 암호를 어떻게 상대방에게 전달하죠? 암호 자체를 보내는 것도 누군가 가로챌 수 있잖아요!

딜레마 발생!

비밀 통신을 하려면 먼저 "우리의 비밀 열쇠는 OO야"라고 말해야 하는데, 그것마저 누가 가로채면 끝장입니다. 이걸 "키 교환 문제 (Key Exchange Problem)"라고 부릅니다.

수십 년간 암호학자들을 괴롭힌 문제였습니다. 어떻게 해결했을까요?

3. 해결책 - 상자와 열쇠

아이디어: 잠그는 것과 여는 것을 분리하자

특수한 상자를 상상해보세요:

  • 이 상자는 누구나 뚜껑을 닫을 수 있습니다 (딸깍 잠김)
  • 하지만 열 수 있는 열쇠는 나만 가지고 있습니다
나 = 쇼핑몰 서버 1. 나는 특수한 상자를 만들어서 전 세계에 뿌림 "나한테 비밀 보내고 싶은 사람! 이 상자에 넣고 닫아서 보내!" 2. 당신(고객)이 카드번호를 그 상자에 넣고 딸깍 닫아서 보냄 3. 중간에 해커가 가로채도 → 열쇠가 없으니 못 열어요 4. 나만 열쇠가 있으니까 → 나만 열어서 카드번호를 볼 수 있어요

핵심 포인트

"잠그는 것"과 "여는 것"에 서로 다른 키를 씁니다.

잠그는 키(상자) = 공개키 → 공개해도 OK

여는 키(열쇠) = 개인키 → 나만 비밀로 보관

이 방식을 비대칭키 암호화 또는 공개키 암호화라고 부릅니다. 처음 보는 사람과도 비밀 열쇠를 미리 공유하지 않고 안전하게 통신할 수 있게 됐습니다!

4. 새로운 문제 - 가짜 상자?

질문

누군가 "나 A쇼핑몰이야~ 여기 내 상자!"라고 상자를 보내왔습니다. 그런데... 그 상자를 보낸 게 진짜 A쇼핑몰인지 어떻게 알죠? 해커가 "나 A쇼핑몰이야~" 하고 자기 상자를 보냈을 수도 있잖아요!

중간자 공격 (Man-in-the-Middle Attack)

해커가 중간에서 가짜 상자를 보내면, 당신은 해커의 상자에 카드번호를 넣게 됩니다. 해커가 열쇠를 갖고 있으니까 털리는 거죠.

해결책: 신분증 시스템

힌트 질문

현실에서 은행에 가면 "본인 맞으세요?"라고 물어봅니다. 이때 뭘 보여주나요? 그리고 그게 왜 믿을 만한가요?

답

신분증을 보여주죠. 이건 국가가 '공인'한 거니까 은행도 이걸 보면 인증이 됩니다.

국가라는 믿을 수 있는 제3자가 "이 사람이 진짜 김OO이 맞아"라고 보증해주는 거죠.

인터넷에서도 똑같습니다

현실 세계 인터넷 세계
국가 인증기관 (CA, Certificate Authority)
신분증 인증서 (Certificate)

예를 들어 Let's Encrypt, DigiCert, Comodo 같은 회사들이 CA입니다. 이들이 "이 공개키(상자)는 진짜 A쇼핑몰 거야"라고 도장 찍어서 인증서를 발급해줍니다.

5. 신뢰의 시작점 - 루트 인증서

질문

그런데 "발급한 것"도 진짜인지 가려야 하잖아요. 브라우저는 어떻게 "이 인증서가 진짜 믿을 만한 CA가 발급한 건지" 알 수 있나요?

현실에서 신분증을 생각해보세요. "이게 진짜 정부가 발급한 건가?"를 매번 확인하나요?

사실 우리는 태어날 때부터 "주민등록증은 이렇게 생겼다", "정부 마크는 이거다"라고 이미 알고 있습니다. (배우거나, 사회가 알려주거나)

인터넷도 마찬가지

브라우저(크롬)나 운영체제(윈도우)를 설치할 때, 이미 "믿을 수 있는 CA 목록"이 내장되어 있습니다.

"이 CA들의 도장은 진짜야"라고 처음부터 심어놓은 거예요.

이걸 루트 인증서 (Root Certificate)라고 부릅니다.

전체 흐름

1. 쇼핑몰이 CA한테 "나 진짜 A쇼핑몰이야" 증명하고 인증서 받음 2. 당신이 쇼핑몰 접속하면, 쇼핑몰이 인증서를 보여줌 3. 브라우저가 확인: "이 인증서에 찍힌 CA 도장이... 내 목록에 있네? OK 믿을게!"

6. 실제 동작 - 두 방식의 조합

아까 두 가지 암호화 방식을 배웠습니다:

방식 장점 단점
대칭키 (둘만 아는 암호) 빠름 키 공유가 문제
비대칭키 (상자와 열쇠) 키 공유 문제 해결 느림 (수학 계산 복잡)

질문

그러면 비대칭키만 쓰면 되겠네요?

비대칭키는 엄청 느립니다. 웹사이트에서 이미지, 동영상, 수많은 데이터를 주고받을 때마다 이 느린 방식을 쓰면... 인터넷이 거북이가 됩니다.

해결책

처음에 비대칭키로 키를 공유하고, 그 다음부터 대칭키를 쓰면 됩니다!

실제 TLS 핸드셰이크

1처음 연결 - 느린 비대칭키 사용

서버가 인증서(공개키 포함)를 보냄

2인증서 검증

브라우저가 인증서 확인 (CA 도장 확인)

3비밀키 공유

브라우저가 "임시 비밀키"를 만들어서 공개키로 암호화해 보냄

4비밀키 획득

서버가 개인키로 열어서 "임시 비밀키"를 얻음 → 이제 둘 다 같은 비밀키를 가짐!

5이후 통신 - 빠른 대칭키 사용

이 비밀키로 모든 데이터를 암호화해서 주고받음

7. 브라우저의 자물쇠 아이콘

브라우저 주소창에 있는 자물쇠 아이콘은 두 가지를 의미합니다:

체크 의미
✓ 이 사이트의 인증서가 유효해 (진짜 그 사이트 맞아)
✓ 지금 통신이 암호화되어 있어 (중간에 못 훔쳐봄)

자물쇠가 보이면

"당신이 접속한 사이트가 가짜가 아니고, 주고받는 데이터를 아무도 못 엿본다"는 뜻입니다.

8. SSL이란 정확히 무엇인가?

SSL (Secure Sockets Layer)은 지금까지 배운 이 모든 과정을 정의한 규약(프로토콜)입니다.

  • 인증서 어떻게 검증할지
  • 키 교환 어떻게 할지
  • 암호화 어떻게 할지

이런 약속/규칙의 묶음이에요.

SSL vs TLS

버전 상태
SSL 1.0, 2.0, 3.0 옛날 거, 보안 취약, 사용 안 함
TLS 1.2, 1.3 현재 사용하는 것

사실 SSL은 옛날 이름입니다. 지금은 TLS를 쓰지만, 사람들이 습관적으로 "SSL"이라고 부릅니다. "SSL 인증서"처럼요.

자주 쓰는 표현

표현 의미
"SSL 인증서를 발급받다" CA에서 인증서 받기
"SSL 인증서를 설치하다" 서버에 인증서 적용하기
"SSL을 적용하다" 사이트에 HTTPS 설정하기
"이 사이트는 SSL이 적용되어 있다" HTTPS로 암호화 통신 중

실무에서

서버 세팅할 때 "SSL 인증서 설치한다"라고 하면, 결국 "우리 서버도 저 자물쇠 뜨게 해서 안전한 HTTPS 통신 가능하게 만든다"는 뜻입니다.

최종 요약

SSL/TLS 한눈에 보기

1. 암호화 - 둘만 아는 규칙으로 내용을 섞어서 보낸다

2. 키 교환 문제 - 처음 만난 사람과 어떻게 비밀 규칙을 공유하지?

3. 공개키/개인키 - 잠그는 키는 공개, 여는 키는 비밀로!

4. 인증서 - CA가 "이 공개키는 진짜 그 사이트 거야"라고 보증

5. 루트 인증서 - 브라우저에 미리 "믿을 CA 목록" 내장

6. 하이브리드 - 처음엔 느린 방식으로 키 공유, 이후엔 빠른 방식으로 통신

결론: SSL/TLS = 이 모든 과정을 정의한 프로토콜
HTTPS = HTTP + SSL/TLS (안전한 웹 통신)
